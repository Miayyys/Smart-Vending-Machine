# 校园智能柜 — 服务器端介绍

## 1. 概述

部署于阿里云轻量服务器，通过 Docker Compose 一体化编排 6 个服务，提供 MQTT 消息中间件、业务 API、AI 推理、固件分发和运营管理后台。三端（硬件设备、手机小程序、管理后台）均通过本服务通信。

| 服务 | 镜像/技术 | 端口 | 用途 |
|------|----------|------|------|
| EMQX | emqx:5.8 | 1883 (MQTT) | 设备/前端消息中间件 |
| MySQL | mysql:8.0 | 3306 (内网) | 业务数据存储 |
| Backend | Spring Boot 3.2 + Java 17 | 8080 | REST API + MQTT 消息处理 |
| Inference | Python FastAPI + ONNX Runtime | 8000 | 服务端 AI 商品识别 |
| Nginx | nginx:1.27-alpine | 80/443 | 反向代理 + 静态资源 |
| Frontend | 原生 JS SPA | — | 运营管理后台，嵌入 Nginx |

内存限制：EMQX 450MB + MySQL 400MB + Backend 400MB + Inference 512MB + Nginx 32MB，适配 1.6GB 物理内存。

## 2. 核心业务功能

### 2.1 开门购物（与硬件 + 小程序配合）

```
手机小程序 POST /api/mobile/open {userId, deviceId}
  → Backend 生成 batchId，存储 userId 关联到 MobileUserSession
  → Backend 通过 MQTT 向设备下发 unlock 指令: retail/D01-F1/cmd/unlock
  → 设备开锁，用户取货，关门
  → 设备拍照并上传 (POST /api/snapshot/upload，token 鉴权，免 Basic Auth)
  → Backend 异步调用 Inference 服务做 AI 识别
  → 开门照: 记录 count_before，不动库存
  → 关门照: 找最近一次 OPEN → 计算 diff = before - after → 生成订单 (UNPAID)
           → 更新库存为关门后数量 → MQTT 推送通知前端刷新
  → 小程序轮询 /api/mobile/order/{batchId} 等待订单生成
  → 用户支付 POST /api/mobile/pay → 订单状态变为 PAID
```

### 2.2 拍照上传与 AI 识别

设备通过 HTTPS multipart 上传图片到 `/api/snapshot/upload`，携带 token（设备专用，存入 device_token 表）、deviceId、floor、doorAction（OPEN/CLOSE/SYNC）。

- **token 校验**：匹配 device_token 表，且必须与请求 deviceId 一致（防跨设备伪造）
- **存储**：`/app/snapshots/{ts}_{deviceId}_F{floor}_{action}.jpg`，入库 snapshots 表
- **自动清理**：每设备保留最近 200 张（`SNAPSHOT_KEEP` 配置），超出删最旧记录和文件
- **识别触发**：上传成功后异步调用 Inference 服务的 `/detect` 接口（ONNX YOLO11n 模型，score≥0.25，NMS≥0.45），识别结果写入 snapshot 的 detection_data 字段

类别映射（服务端与设备端一致）：nongfu(农夫山泉)→productId 6, runtian(润田)→7, soda(可乐)→2, yogurt(酸奶)→8。注意：yogurt 误检率高，推理服务设独立阈值 0.60。

### 2.3 库存管理

- 库存表 stock (deviceId + productId 联合主键)，记录当前数量和阈值
- **关门后自动更新**：直接设置各商品数量为关门识别结果
- **补货告警**：库存低于阈值 (默认 3) 自动生成 restock_alert
- 提供 `/api/devices/{id}/stock`（查询库存）、`/api/stock/low`（低库存清单）、`/api/stock/reconcile`（手动校准）

### 2.4 订单与结算

- 订单表 order_info：batch_id、userId、totalAmount、payStatus (UNPAID/PAID/FAILED)、createTime、settleTime
- **自动生成**：关门识别后，diff > 0 的商品自动创建 UNPAID 订单
- **支付**：用户调用 `/api/mobile/pay`，校验 userId 匹配，更新为 PAID
- 手机端 `/api/mobile/orders?userId=` 查历史订单，`/api/mobile/order/{batchId}` 查单笔

### 2.5 异常告警

- 设备通过 MQTT `retail/{device}/event/anomaly` 上报异常（如强制开门、开门超时、拍照失败）
- Backend 入库 anomaly_alert 表，管理后台可查看和处理
- 服务器可通过 MQTT `cmd/err` 向设备下发异常（如双模比对不一致），设备进入 ERROR 状态亮红灯

### 2.6 固件与模型分发

- 管理后台 `/api/firmware/upload` 上传固件/模型文件，记录版本号和 MD5
- 设备通过 HTTPS Basic Auth 下载 `/api/firmware/{id}/download`
- 支持 OTA 推送：服务器通过 MQTT `cmd/ota` 发送下载 URL，设备自行下载并切换分区

## 3. MQTT 消息体系

### 主题约定

| 主题 | 方向 | 说明 |
|------|------|------|
| `retail/{device}/cmd/unlock` | 下行 | 开门指令 |
| `retail/{device}/cmd/lock` | 下行 | 锁门指令 |
| `retail/{device}/cmd/reboot` | 下行 | 重启指令 |
| `retail/{device}/cmd/sync` | 下行 | 触发同步拍照 |
| `retail/{device}/cmd/ota` | 下行 | OTA 升级（含 url、type） |
| `retail/{device}/cmd/err` | 下行 | 服务器下发异常 |
| `retail/{device}/event/detect` | 上行 | 设备端识别结果上报 |
| `retail/{device}/event/anomaly` | 上行 | 设备端异常上报 |
| `retail/{device}/event/ota_progress` | 上行 | OTA 进度上报 |
| `retail/{device}/telemetry/weight` | 上行 | 重量遥测（预留） |

### 账号鉴权

EMQX 启用 built_in_database 认证器，两对账号分离 UI 端和设备端权限：

| 账号 | 用途 |
|------|------|
| retail_ui | 管理后台、后端服务 |
| retail_device | 硬件设备 |

密码存储于 `.env` 文件，通过 `emqx/init-users.sh` 写入 EMQX。

## 5. 运营管理后台

单页应用（原生 JS），嵌入 Nginx 静态服务。功能模块：

- **仪表盘**：设备在线状态、今日订单数、库存概览
- **商品管理**：商品列表、价格
- **库存监控**：实时库存、低库存告警、补货确认
- **订单管理**：订单列表、支付状态
- **抓拍记录**：按设备/楼层/开关门筛选、查看照片、查看识别结果
- **异常告警**：异常列表、处理状态
- **设备控制**：远程开门/锁门/重启/同步
- **固件管理**：上传、列表、下载

前端通过 MQTT over WebSocket (`wss://.../mqtt`) 接收实时推送，库存变化和异常告警即时刷新。

## 6. AI 推理服务

独立容器，ONNX Runtime 替代 PyTorch，内存占用约 200MB（vs 800MB+）。提供 `/detect` 接口：

- 输入：图片文件路径
- 输出：各类商品数量 (counts)、总件数 (total)、每件详情 (类别/置信度/坐标)、处理耗时
- 模型：YOLO11n, ONNX 格式，320×320 输入
- 配置 4 类独立置信度阈值，yogurt 提高至 0.60 以减少误检
- 管理后台 `/api/model/` 路径反向代理至推理服务，支持模型切换（演示用）

## 7. 部署与运维

```bash
cd ~/retail-cloud
docker compose up -d --build     # 首次启动（约 3-5 分钟）
docker compose ps                # 查看服务状态
docker compose logs -f backend   # 跟踪后端日志
docker compose down              # 停止（保留数据）
docker compose down -v           # 完全重置（删除数据卷）

# 首次启动后必须执行一次：启用 EMQX 账号鉴权
docker compose exec emqx sh /opt/emqx/init-users.sh
```

公网已放行 1883 (MQTT) 和 443 (HTTPS)，3306 和 18083 仅本机访问。nginx 整站 Basic Auth（凭证见 `.htpasswd`），设备上传接口豁免。

数据位置：MySQL 数据绑定到 `~/retail-cloud/mysql/data`，固件 `/app/firmware`，照片 `/app/snapshots`。
