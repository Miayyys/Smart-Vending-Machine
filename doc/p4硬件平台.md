# 智能售货机 — 硬件平台介绍

## 1. 平台总览

基于 ESP32-P4 的边缘 AI 零售终端。用户在手机端扫码开门、选取商品、关门即自动结算——整套流程由设备本地 YOLO 视觉识别驱动。

| 组件 | 型号 | 用途 |
|------|------|------|
| 主控 | 微雪 ESP32-P4-Module | 双核 360MHz, 8MB PSRAM, 16MB Flash |
| 协处理器 | ESP32-C6 | WiFi 6 + BLE 5, 独立承担 MQTT/HTTPS 通信 |
| 摄像头 | OV5647 | MIPI CSI, 800×800 RAW8, ISP 片上转 RGB565 |
| 门磁 | MS-1 干接点 | 检测柜门开关 |
| 舵机 | SG90 | 0°锁门 / 90°开锁, 50Hz PWM |
| 指示灯 | RGB 共阳 3 路 | 5 色状态指示 |

WiFi 凭证持久化在 NVS，可在运行时通过串口命令动态更新，无需重新烧录。

---

## 2. 核心特色

**端侧 AI 识别**：所有商品识别在 ESP32-P4 本地完成，YOLO11n 模型经 INT8 量化后部署在设备上，单次推理即可输出柜内商品的位置、类别和置信度。图片无需上传服务器做识别，节省带宽和算力成本。

**双槽 OTA 无感升级**：固件和 AI 模型各有独立双槽分区。新版本下载到非活跃槽，切换标记后重启即生效。升级失败不影响当前运行版本，模型可独立升级无需重新烧录固件，运营方可随时更新商品识别种类。

**多层异常检测**：设备本地监控门磁状态、开门超时、摄像头故障；服务器端比对开/关门照片检测异物放入，并可选启用服务端识别与端侧结果交叉验证。所有异常实时通过 MQTT 上报。

## 3. 购物流程

一次完整的购物流程如下：

1. 用户手机扫码，App 向服务器请求开门
2. 服务器通过 MQTT 向设备下发 `unlock` 指令（含 `duration` 字段，默认 30s）
3. 设备收到指令：舵机开锁 (90°)，状态灯由蓝变青，门磁开始监听开关门
4. 用户拉开柜门 → 门磁检测开门 → 状态灯变绿，自动触发：拍照 (800×800 JPEG) → YOLO 本地识别 → 照片 HTTPS 上传（标注 "OPEN"，附带设备ID/楼层/批次号）→ 识别结果 MQTT 上报 (JSON: 每个商品的类别、置信度、边界框坐标)
5. 用户取出或放回商品（设备保持绿灯，此期间若收到关门事件则入队等待）
6. 用户关上柜门 → 门磁检测关门 → 舵机立即锁门 (0°)，状态灯变黄，自动触发：再次拍照 + YOLO 识别 → 上传（标注 "CLOSE"，同一批次号）→ 上报识别结果 → 状态灯恢复蓝色待机
7. 服务器比对开关门两次商品差异，结算扣费，App 显示购物清单

**设计要点**：
- 关门瞬间先锁舵机再拍照，防止拍照期间再次开门导致计数不一致
- 开关门共用 `batch_id`，服务器据此关联同一笔购物
- UNLOCKED 状态下 30s 未开门自动锁门，8s 未开门自动拍照（防止解锁后长期无人操作）
- SYNC 命令可随时触发后台拍照同步，不改变门锁状态，用于远程盘点

## 4. 异常处理

| 异常场景 | 触发条件 | 严重级别 | 设备行为 |
|----------|----------|----------|----------|
| 强制开门 | LOCKED 状态下门磁检测到开门 | CRIT | 红灯 5s + MQTT 上报 |
| 开门超时 | 开门后超过 60s 未关门 | WARN | MQTT 上报, 不影响后续正常关门 |
| 摄像头故障 | 拍照失败 | WARN | 红灯 5s + MQTT 上报 |
| 服务器下发的异物检测 | 关门照中未被拿取区域与开门照不一致 | CRIT | 服务器通过 MQTT `cmd/err` 下发, 设备进入 ERROR |
| 双模交叉验证 | 服务器端识别结果与端侧上报不一致 | CRIT | 同上 |

所有异常均通过 MQTT `retail/{ID}/event/anomaly` 上报，格式 `{"type":"...","level":"WARN|CRIT"}`。ERROR 状态下红灯亮 5 秒后自动恢复至异常前状态。服务器可随时下发异常让设备进入 ERROR 状态。

## 5. 状态指示

设备通过 RGB 三色灯直观展示当前状态：

| 灯光 | 状态 | 说明 |
|------|------|------|
| 蓝 | LOCKED | 柜门锁闭，待机中 |
| 青 | UNLOCKED | 舵机已开锁，等待用户开门 |
| 绿 | DOOR_OPEN | 门已开，用户正在选购 |
| 黄 | ANALYZING | 正在拍照/识别/上传 |
| 红 | ERROR | 异常，5s 后自动恢复 |

---

## 6. 状态机

```
LOCKED(蓝) → UNLOCKED(青) → DOOR_OPEN(绿) → ANALYZING(黄) → LOCKED
    ↑                                                              │
    └──────────────────────────────────────────────────────────────┘
                        任意状态 ──▶ ERROR(红) ──5s自动恢复──▶ 异常前状态
```

| 状态 | 舵机 | 门磁 | 超时 |
|------|------|------|------|
| LOCKED | 锁闭 | 忽略 | — |
| UNLOCKED | 开锁 | 监听 | 30s 无操作自动锁门 |
| DOOR_OPEN | 开锁 | 监听 | 60s 未关门上报异常 |
| ANALYZING | — | 事件排队 | 队列中事件 15s TTL |
| ERROR | — | — | 5s 后自动恢复 |

状态转换受白名单约束，防止非法跳转。拍照/识别/上传等重任务在独立 FreeRTOS 任务 (`work_task`, 栈 12KB) 中执行，通过信号量触发。

## 7. GPIO 引脚分配

| GPIO | 连接 | 配置 |
|------|------|------|
| 2 | 门磁 MS-1 | 输入, 内部上拉, 任意边沿中断 |
| 6 | RGB 红灯 | 输出, 共阳 (0=亮) |
| 7 | OV5647 I2C SDA | 开漏, I2C0 |
| 8 | OV5647 I2C SCL | 开漏, I2C0 |
| 20 | RGB 蓝灯 | 输出, 共阳 |
| 21 | RGB 绿灯 | 输出, 共阳 |
| 22 | SG90 舵机 | 输出, LEDC Timer1/Ch1, 50Hz PWM |

舵机使用 Timer1/Ch1，与相机 XCLK 占用的 Timer0/Ch0 互不冲突。

## 8. 摄像头

- OV5647 MIPI CSI 2-lane, 800×800 RAW8 @50fps
- 管线: MIPI → ISP 色彩处理 (RAW8→RGB565) → PPA 硬件旋转 (90°+水平镜像) → JPEG 硬件编码 (Q=80)
- YOLO 走独立路径：`captureRawRgb565()` 直接获取原始 RGB565，不经旋转和压缩

---

## 9. YOLO 端侧推理

- 模型: YOLO11n, INT8 量化, 输入 320×320 RGB565, 从 Flash 分区加载 `.espdl` 文件
- 后处理: 三检测头 stride {8,16,32}, DFL 解码 + 量化域 score 阈值 + NMS
- 检测参数: score≥0.6, NMS≥0.7, 最多 10 个目标
- 类别: soda(可乐) / nongfu(农夫山泉) / runtian(润田) / yogurt(酸奶)
- 模型双槽 (model_0 / model_1), `model_otadata` 标记激活槽, 重启自动切换

---

## 10. 门磁传感器

干簧管磁控原理：磁体靠近(关门)→触点闭合→GPIO2 拉到 GND；磁体远离(开门)→触点断开→GPIO2 被内部上拉至高电平。

防抖流程：GPIO 任意边沿中断 → ISR 50ms 窗口过滤 → 后台任务排空队列只留最新事件 → 延迟 30ms 等触点稳定 → 51 次采样 (间隔 2ms) 多数表决 → 低电平占比 ≥70% 判关门。干接点浮空噪声约 50% 高低比例，机械闭合后稳定低电平 (~100%)，70% 阈值能可靠区分。

## 11. OTA 升级

16MB Flash 双槽设计：

| 分区 | 大小 | 说明 |
|------|------|------|
| app_0 | 4MB | 固件主槽 |
| app_1 | 4MB | 固件备槽 |
| model_0 | 4MB | 模型主槽 |
| model_1 | 4MB | 模型备槽 |
| model_otadata | 4KB | 模型激活槽标记 |

- **固件 OTA**: MQTT 收到指令 → HTTPS Basic Auth 下载 → 写入非活跃 app 分区 → `esp_ota_set_boot_partition` → 重启。启动失败时 bootloader 自动回退。
- **模型 OTA**: MQTT 收到 `{"type":"model","url":"..."}` → HTTPS 下载到非活跃 model 分区 → 写入 `model_otadata` 一字节标记 → 重启。`yolo_init()` 读标记自动加载新槽。

下载过程通过 MQTT `event/ota_progress` 实时上报进度百分比。

---

## 12. 事件通信

所有模块通过 `esp_event` 事件总线解耦。主要 MQTT 主题：

| 方向 | 主题 (相对 `retail/{ID}/`) | 内容 |
|------|---------------------------|------|
| 下行 | `cmd/#` | unlock / lock / reboot / sync / ota / err |
| 上行 | `event/detect` | 识别结果: batch_id, door_action, detections[] |
| 上行 | `event/anomaly` | 异常: type, level |
| 上行 | `event/ota_progress` | OTA 进度: type, status, progress, message |

图片通过 HTTPS multipart/form-data POST 到 `/api/snapshot/upload`，含 token / deviceId / floor / doorAction / file 五个字段。

## 13. 串口调试

UART0, 115200bps, 提示符 `p4>`。

| 命令 | 功能 |
|------|------|
| `photo` | 手动触发拍照→YOLO→上传→上报完整流程 |
| `info` | 查看设备ID、摄像头、网络、状态等信息 |
| `scan` | I2C 总线设备扫描 |
| `state` | 查看状态机当前状态 |
| `door_open` / `door_close` | 模拟门事件 |
| `wifi <SSID> [密码]` | 切换 WiFi 并持久化到 NVS |
| `servo <0-180>` | 手动控制舵机 |
| `yolo_test` | 对嵌入的 test.jpg 运行 YOLO 推理 |

## 14. 构建

```bash
. ~/esp/esp-idf/export.sh
cd P4HarfWare/platform1
idf.py build && idf.py flash monitor
```

- 目标: ESP32-P4 (工程样品 pre-v3, CPU 360MHz)
- 依赖: ESP-IDF v5.4 + ESP-DL
- 模型训练: YOLO11n → ONNX → ESP-DL INT8 量化 → `.espdl`
