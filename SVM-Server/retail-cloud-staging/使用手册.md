- **上传（设备 → 云）**：MQTT 上报事件/遥测 + HTTP 上传抓拍图片。
- **下发（云 → 设备）**：MQTT `cmd` 主题下发指令 / OTA。

1. 浏览器开 `https://<server-ip>`
2. 弹"您的连接不是私密连接"（自签证书）→ "高级" → "仍要继续访问"

## 3. MQTT 主题规约（**核心，先读这段**）

前缀固定 `retail`，三层结构 `retail/<device>/<kind>/<sub>`。

### 3.1 上传方向（设备 → 云）

| Topic                           | Payload（JSON）                                    | 后端动作                                          |
| ------------------------------- | ------------------------------------------------ | --------------------------------------------- |
| `retail/<dev>/event/take`       | `{"pid":1,"act":"TAKE","qty":1,"w":550}`         | 取货事件：扣库存、入 `take_event`、低库存则建 `restock_alert` |
| `retail/<dev>/event/settle`     | `{"uid":"user01","amount":9.50,"status":"PAID"}` | 结算：`order_info` 建订单（自动查价补金额）                  |
| `retail/<dev>/event/anomaly`    | `{"type":"DOOR_FORCED","level":"WARN"}`          | 异常告警入 `anomaly_alert`                         |
| `retail/<dev>/telemetry/weight` | `{"w":550}`                                      | 称重遥测（仅记日志，不落业务库）                              |

**take 字段说明（字段名别写错！）**：

- `pid`（**不是** `productId`）——商品 ID
- `act` —— 动作，默认 `TAKE`
- `qty` —— 数量，默认 1
- `w` —— 重量(克)，可空

**settle 字段**：`uid`（用户ID）、`amount`（金额，可省，省则后端自动算）、`status`（默认 `PAID`）。
**anomaly 字段**：`type`、`level`（默认 `WARN`），其余字段会整包存进告警详情。

### 3.2 下发方向（云 → 设备）

| Topic                      | Payload（JSON）                                                                     | 含义     |
| -------------------------- | --------------------------------------------------------------------------------- | ------ |
| `retail/<dev>/cmd/restock` | `{"action":"RESTOCK_DONE","items":[{"productId":1,"qty":10}]}`                    | 补货完成   |
| `retail/<dev>/cmd/unlock`  | `{"action":"UNLOCK","duration":30}`                                               | 远程开门   |
| `retail/<dev>/cmd/lock`    | `{"action":"LOCK"}`                                                               | 远程锁门   |
| `retail/<dev>/cmd/sync`    | `{"action":"SYNC_STOCK"}`                                                         | 同步库存   |
| `retail/<dev>/cmd/reboot`  | `{"action":"REBOOT"}`                                                             | 重启设备   |
| `retail/<dev>/cmd/price`   | `{"action":"UPDATE_PRICE","productId":1,"price":2.50}`                            | 更新价格   |
| `retail/<dev>/cmd/ota`     | `{"action":"OTA_UPDATE","version":"v1.2.3","url":"...","md5":"...","size":12345}` | OTA 升级 |

> 下发的 `cmd/*` 后端**不消费**，只由设备订阅。云端的下发由前端"设备控制/OTA升级"页通过 `wss://.../mqtt`（MQTT.js over WebSocket）发起。设备端要订阅 `retail/+/cmd/#` 或 `retail/<本机id>/cmd/#` 才能收到。

### 3.3 云端发出的实时通知（云 → 前端）

| Topic                         | 含义                                                   |
| ----------------------------- | ---------------------------------------------------- |
| `retail/<dev>/event/snapshot` | 抓拍图上传成功后，后端 `MqttPublisher` 发给前端即时刷新拍照记录页，无需等 30s 轮询 |

前端订阅的是 `retail/+/event/#`，所以 take/settle/anomaly/snapshot 全能收到。注意 `createTime` 字段后端已转成字符串（Jackson 不认 LocalDateTime，踩过坑）。

## 4. MQTT 设备鉴权与连接

### 4.1 三个账号

| 账号              | 密码               | 用途                               |
| --------------- | ---------------- | -------------------------------- |
| `retail_device` | `<mqtt-device-password>`   | **设备端**用，1883 连 broker 上报        |
| `retail_ui`     | `<mqtt-ui-password>`    | 前端 / 后端用（前端走 WebSocket，后端走 1883） |
| `admin`         | `<emqx-dashboard-password>` | 仅 EMQX Dashboard，不用于 MQTT 连接     |

EMQX 已启用 `password_based:built_in_database` 内置鉴权，匿名发布会被拒。

### 4.2 设备端连哪里？

- **真单片机/模拟器**：`tcp://<server-ip>:1883` —— ⚠️ **目前 1883 只绑 127.0.0.1，阿里云安全组未放行**。所以**外部真实设备/本机 mqtt.fx 暂时连不上 1883**。两种解法：
  - **A（推荐联调）**：在**服务器上**跑模拟客户端（ssh 上去发），走 127.0.0.1:1883。
  - **B（要外网真设备）**：放行 1883。先在阿里云安全组放 1883，再改 `docker-compose.yml` 把 `127.0.0.1:1883:1883` 改成 `0.0.0.0:1883:1883`（或去掉 IP 前缀）后 `docker compose up -d emqx`。放行前务必确认 EMQX 鉴权账号已生效（已生效）。
- **前端**：走 `wss://<server-ip>/mqtt`（nginx 代理到 emqx:8083 WebSocket），带 Basic Auth，已通。

## 5. 上传指令怎么测？

> "上传指令"= 模拟设备向云端发 MQTT event。云端的"上报"全部走 MQTT 1883（不是 HTTP）。

### 5.1 种子数据（直接拿来用）

- 设备：`D01`（1号柜，教学楼A栋1层）、`D02`（2号柜，图书馆2层）
- 商品：1=矿泉水、2=可乐、3=薯片、4=面包、5=咖啡（价格 2.00~6.00）
- 抓拍 token：`<snapshot-token-D01>`（绑D01）、`<snapshot-token-D02>`（绑D02）

### 5.2 方式一：EMQX Dashboard WebSocket 客户端（**服务器内**，最方便，免装东西）

这就是你记忆里"后端的虚拟设备"——**EMQX Dashboard 自带的 WebSoxket MQTT 客户端**，之前用它替代 mosquitto。

1. `ssh user@<server-ip>` 上服务器

2. 端口转发（本机 SSH 配置已就绪）：
   
   ```
   ssh -L 18083:127.0.0.1:18083 myserver -N
   ```
   
   （保持窗口开着；或用 VSCode Remote 的端口转发功能转发 18083）

3. 本机浏览器开 `http://127.0.0.1:18083` → 登录 `admin / <emqx-dashboard-password>`

4. 左侧菜单 → **问题分析 / WebSockets**（或 "Tools" / "WebSocket Client"，不同版本叫法略异）

5. 连接信息：
   
   - Broker：`127.0.0.1`，端口 `1883`，协议 MQTT
   - 用户名 `retail_device`，密码 `<mqtt-device-password>`
   - Client ID 随意（如 `emu_D01`）
   - 连接

6. **订阅** `retail/D01/cmd/#`（接收下发指令，便于观察云→设备）

7. **发布**上传事件，举三个标准例子：
   **取货（扣库存）：**
   
   - Topic：`retail/D01/event/take`
   - Payload：`{"pid":1,"act":"TAKE","qty":1,"w":550}`
   
   **结算（建订单）：**
   
   - Topic：`retail/D01/event/settle`
   - Payload：`{"uid":"user01","amount":9.50,"status":"PAID"}`
   
   **异常告警：**
   
   - Topic：`retail/D01/event/anomaly`
   - Payload：`{"type":"DOOR_FORCED","level":"WARN"}`

8. 发布后回到前端 `https://<server-ip>` 库存/订单/告警页，应**实时刷新**（MQTT 推送 + 30~60s 轮询兜底）。

> 服务器上也可用 `mosquitto_pub`（服务器已装）一行发：
> 
> ```
> mosquitto_pub -h 127.0.0.1 -p 1883 -u retail_device -P '<mqtt-device-password>' -t 'retail/D01/event/take' -m '{"pid":1,"qty":1,"w":550}'
> ```

### 5.3 方式二：MQTT.fx（本机 GUI 模拟器，**需先放行 1883**）

MQTT.fx 是桌面版 MQTT 客户端。要在本机连**公网** broker，必须先按 §4.2-B 放行 1883。放行后：

1. 新建 Connection：
   - Broker Address：`<server-ip>`，Port `1883`
   - User Name：`retail_device` / Password：`<mqtt-device-password>`
   - Client ID 随意
2. Connect → 绿灯
3. Subscribe 标签页订阅 `retail/D01/cmd/#`（看下发）
4. Publish 标签页发上面三条上传事件（topic + payload 一致）

> 没放行 1883 也能用 MQTT.fx，但要把 broker 改成服务器内网的 127.0.0.1——这只能在服务器图形界面里操作，一般不在本机用。本机用 MQTT.fx 默认就是冲外网 1883，所以**必须先放行**。

### 5.4 方式三：真实单片机（ESP32/STM32+WiFi 等）

设备端逻辑要点（伪代码）：

```c
// 1. 连 WiFi 后连 MQTT
mqtt.connect("tcp://<server-ip>:1883", clientId,
            username="retail_device", password="<mqtt-device-password>");

// 2. 订阅下发指令（云端→设备）
mqtt.subscribe("retail/D01/cmd/#");   // 收到后解析 JSON 执行 UNLOCK/LOCK/REBOOT/OTA_UPDATE...

// 3. 开门事件 → 上报取货（扣库存）
mqtt.publish("retail/D01/event/take",
  "{\"pid\":1,\"act\":\"TAKE\",\"qty\":1,\"w\":550}", QOS=1);

// 4. 关门事件 → 上报结算（建订单）
mqtt.publish("retail/D01/event/settle",
  "{\"uid\":\"card_0001\",\"amount\":9.50,\"status\":\"PAID\"}", QOS=1);

// 5. 检测到异常 → 上报
mqtt.publish("retail/D01/event/anomaly",
  "{\"type\":\"DOOR_FORCED\",\"level\":\"WARN\"}", QOS=1);

// 6. 楼层抓拍图：走 HTTP 上传（见 §7），不是 MQTT
```

**单片机接 OTA/Lock 的响应约定**（设备自己解析 payload 里的 `action` 字段）：

- `{"action":"UNLOCK","duration":30}` → 开门 30 秒
- `{"action":"LOCK"}` → 锁门
- `{"action":"REBOOT"}` → 软重启
- `{"action":"RESTOCK_DONE","items":[...]}` → 补货完成后可回发一次 `event/take` 的反向同步，或自己上报 settle 归零（按业务定）
- `{"action":"OTA_UPDATE","url":"...","md5":"..."}` → HTTP GET url，校验 md5，刷写，成功后可上报 `event/anomaly type=OTA_SUCCESS` 之类（按业务定，**当前后端不强制 OTA 回执**）

> 真实单片机也必须先放行 1883（§4.2-B）才能连公网。建议小批量测试时直接 ssh 到服务器用 EMQX Dashboard 联调更省事。

## 6. 下发指令怎么测？

> "下发指令"= 云端向设备发 `cmd` 主题。前端页面已封装好。

### 6.1 前端"设备控制"页（最简单）

1. `https://<server-ip>` → 登录 → Ctrl+F5 → 进"设备控制"
2. 选目标设备（D01/D02）
3. 右侧"快捷指令"点一个 → 自动填好 topic + payload
4. 点"发送指令" → 走 `wss://.../mqtt` 下发
5. 下发历史显示成功/失败

快捷指令清单见 §3.2。也可在左侧手填自定义 topic/payload（JSON）测试。

### 6.2 OTA 下发

1. "OTA升级"页 → 先上传一个固件（`.bin/.hex/.elf`），填版本号
2. 上传成功后列表出现该固件
   3.下半部分"OTA 下发升级"选目标设备 + 固件版本 → 预览指令
3. 发送 → 实际向 `retail/<dev>/cmd/ota` 发 `{"action":"OTA_UPDATE","version":"...","url":"https://.../api/firmware/{id}/download","md5":"...","size":...}`
4. 设备订阅 `cmd/ota` 收到 → 按 url 下载 → md5 校验 → 刷写

> OTA 下载 URL 走公网 HTTPS，设备需带 Basic Auth 凭证下载（`admin/<nginx-basic-auth-password>`），或二期改成设备 token 下载（当前未做）。单片机 HTTP 客户端要带这个 Basic Auth 头。

## 7. 楼层抓拍图片上传（HTTP，**已端到端验证**）

抓拍图**不走 MQTT**，走 HTTP multipart 上传。这是唯一一条对设备开放、**免整站 Basic Auth** 的接口（靠设备专属 token 鉴权）。

### 7.1 上传

```bash
curl -sk -F "token=<snapshot-token-D01>" -F "deviceId=D01" -F "floor=1" \
  -F "doorAction=OPEN" -F "file=@x.jpg" \
  https://<server-ip>/api/snapshot/upload
```

- 字段：`token / deviceId / floor(≥1) / doorAction(OPEN|CLOSE) / file`
- 成功 `200` 返回 `{"id","url","fileName","fileSize","md5","floor","doorAction","createTime"}`
- `floor` 必须 ≥1；`doorAction` 只接受 `OPEN`/`CLOSE`（大小写不敏感）
- 图片存服务器 `~/retail-cloud/snapshots/`，命名 `{ts}_{dev}_F{floor}_{act}.jpg`，MD5 入库
- **自动清理**：每设备保留最近 `SNAPSHOT_KEEP`（默认 200）张，超了删最旧的文件+记录（compose backend env 必须显式传 `SNAPSHOT_KEEP` 否则取不到默认值，**已修**）
- 上传成功后后端发 `retail/<dev>/event/snapshot` 实时通知前端

### 7.2 鉴权矩阵（验证全过）

| 情况                      | 返回                              |
| ----------------------- | ------------------------------- |
| 正确 token + 匹配设备         | 200                             |
| 不带 token                | 401 `missing token`             |
| 错 token                 | 401 `invalid token`             |
| D02 的 token 上传到 D01     | 403 `token not for this device` |
| 缺 floor / doorAction 非法 | 400                             |

### 7.3 查看与删除（受 Basic Auth 保护）

- 列表：`GET /api/snapshot?deviceId=D01&doorAction=OPEN&limit=50`（带 Basic Auth）
- 带楼层筛选：`GET /api/snapshot/search?deviceId=D01&floor=1`
- 图片流：`GET /api/snapshot/{id}/file`（前端 `<img>` 直接显图；浏览器访问目录页需带 Basic Auth）
- 删除：`DELETE /api/snapshot/{id}`
- 看 token（脱敏）：`GET /api/snapshot/token`（运维查当前可用 token 前缀）

前端直接进"拍照记录"页操作，不用手敲 curl。

> 
> ```bash

> ```

## 9. 常用运维命令


## 10. 二期待办 / 已知风险（接手优先看）

1. **认证架构**：REST 无业务鉴权，全靠 nginx Basic Auth 兜底；`frontend/js/mqtt.js` 硬编码 `retail_ui/<mqtt-ui-password>`，F12 可见。→ 后端加登录态、MQTT 凭证移出 JS、或走后端代理 / WebSocket 短期令牌。
2. **自签证书**：浏览器红警告，仅挡"乱闯"，不挡定向攻击。→ 正式证书（或 Let's Encrypt + 域名）。
3. **1883 未公网放行**：外部真实设备/MQTT.fx 连不上 → 需要时按 §4.2-B 放行，配套设备端鉴权账号已就绪。
4. **OTA 下载鉴权**：固件下载 URL 仍走 Basic Auth，单片机 HTTP 客户端需带 `admin` 凭证，安全性差 → 改设备 token 下载。
5. **OTA 无回执**：设备刷写成功/失败后未上报固化协议 → 建议定一个 `event/ota_result` 主题。
6. **`description` 中文乱码**：Windows multipart 中文描述会乱码（不影响功能），可考虑统一 UTF-8 处理。
7. **缓存**：nginx `/js/` 有 `expires 7d`，改前端后必须升缓存戳（当前 `?v=3`）并提示用户 Ctrl+F5。

## 11. 快速联调剧本（一次跑完整条链路）

> 假设已放行 1883 或在服务器内用 EMQX Dashboard。

1. 设备上线：MQTT 连 broker（`retail_device` 账号），订阅 `retail/D01/cmd/#`。
2. **上传·取货**：发 `retail/D01/event/take` `{"pid":1,"qty":1,"w":550}` → 前端"库存管理"D01 矿泉水 -1。若跌破阈值 → "告警管理"出现 PENDING 补货告警。
3. **上传·结算**：发 `retail/D01/event/settle` `{"uid":"user01","amount":9.50,"status":"PAID"}` → "订单管理"出现新订单。
4. **上传·异常**：发 `retail/D01/event/anomaly` `{"type":"DOOR_FORCED","level":"WARN"}` → "告警管理"出现异常记录。
5. **下发·控制**：前端"设备控制"→ 选 D01 → "远程开门"发送 → 设备端 Dashboard 订阅窗口收到 `{"action":"UNLOCK","duration":30}`。
6. **下发·OTA**：前端"OTA升级"上传固件 → 选 D01 + 版本 → 发送 → 设备端收到 `cmd/ota` 含 url/md5 → HTTP 下载 → 校验 → 刷写（回执待二期）。
7. **抓拍上传**：`curl ... /api/snapshot/upload`（带 D01 token）→ 前端"拍照记录"页**即时**出现该图（MQTT `event/snapshot` 推送）。
8. **ACK 补货**：前端"告警管理"点"确认" → 告警状态变 ACK。
9. **手动对齐库存**：前端"库存管理"点某行 → 弹窗改 qty → 实际调 `POST /api/stock/reconcile`（form 参数）→ 库存更新。

跑完 1-9 即覆盖全部上行/下行上传与下发链路。测试数据用完记得清理（库重置种子、订单/事件/告警/抓拍/固件清空），别留脏数据。
